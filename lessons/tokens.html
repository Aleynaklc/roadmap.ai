1 Simple Explanation

Tokenization is the process of breaking raw text into the discrete units that a language model actually processes. A model never sees letters or whole words directly — it sees tokens, which are small chunks of text mapped to integer IDs from a fixed vocabulary. Every LLM interaction starts with tokenization, and every output ends with detokenization (converting token IDs back into text).

Tokens are not always whole words. Modern LLMs use subword tokenization: common words become single tokens ("the" = token 464), rare words split into multiple tokens ("tokenization" = ["token", "ization"]), and individual bytes are the fallback for truly novel characters. The word "unbelievably" might become 3 or 4 tokens depending on the vocabulary.

The dominant algorithm is Byte Pair Encoding (BPE). BPE starts with individual bytes and iteratively merges the most frequently co-occurring pairs into new tokens until a target vocabulary size is reached (typically 30,000 to 100,000 tokens). This gives the vocabulary a natural balance: common words and phrases are single tokens (efficient), while rare sequences decompose into recognizable subword pieces (flexible and robust).

💡 Key insight: The token is the atom of the LLM universe. Understanding tokenization explains why LLMs count letters strangely, why context windows have specific limits, why the same content costs different amounts in different languages, and why certain prompts are more efficient than others. Senior AI engineers think in tokens instinctively.

2 Why It Matters for AI Engineers

Every AI engineering decision involving LLMs is denominated in tokens: API costs are per-token, context window limits are in tokens, rate limits are in tokens-per-minute, and prompt compression strategies operate on tokens. Tokenization also directly affects model behavior: arithmetic is harder because numbers split unexpectedly ("123456" may become ["12", "34", "56"]). Languages with fewer training data representations often have sparser vocabularies, so the same sentence requires 3-5x more tokens than in English — increasing cost and reducing effective context capacity.

3 Real-World Example

OpenAI's tiktoken and the GPT-4 Vocabulary

GPT-4 uses a tokenizer called cl100k_base — a BPE vocabulary of approximately 100,000 tokens trained on internet text and code. The vocabulary has learned that "ing", "tion", "pre", "un", and "ed" are common enough to deserve dedicated tokens. It also learned that entire frequent sequences like " the" (space + the) are single tokens — because the leading space is part of the pattern.

When OpenAI charges for API usage, they count every token in the prompt and every token in the response. A 1,000-word essay in English is roughly 1,300 tokens. The same essay in Thai might be 3,500 tokens — costing 2.7x more and consuming 2.7x more context window. This is why multilingual AI applications require careful token budget management. Enterprise systems use tiktoken to count tokens before sending to the API, implementing hard limits and progressive compression when budgets would be exceeded.

4 Step-by-Step Explanation

How BPE Works (Training Phase)

  1. Initialize with bytes: Start with the 256 possible byte values as the initial vocabulary. Every text in the training corpus is represented as a sequence of bytes.
  2. Count all adjacent pairs: Scan the entire corpus and count how often each adjacent pair of tokens appears together. Example: the pair ("e", "r") might appear 2 million times.
  3. Merge the most frequent pair: Create a new token "er" by merging the top pair. Update all occurrences in the corpus representation.
  4. Repeat: Re-count pairs in the updated corpus, merge the new most frequent pair. Each iteration creates one new token.
  5. Stop at target vocabulary size: After N merge operations, the vocabulary has 256 + N tokens. GPT-4 performs approximately 99,744 merges to reach ~100,000 tokens.

At Inference Time

When new text arrives, the tokenizer applies the learned merge rules in the same order they were created during training — this is deterministic. The resulting token IDs are looked up in the embedding table, converted to vectors, and fed to the Transformer.

5 Python Code Example

python
import tiktoken

# Load GPT-4's tokenizer
enc = tiktoken.encoding_for_model("gpt-4")

# Encode text to token IDs
text = "Tokenization is the foundation of all LLMs."
token_ids = enc.encode(text)
print(f"Token IDs: {token_ids}")
print(f"Token count: {len(token_ids)}")

# Inspect each token boundary
print("
Token breakdown:")
for tid in token_ids:
    piece = enc.decode_single_token_bytes(tid).decode('utf-8', errors='replace')
    print(f"  ID {tid:6d} -> '{piece}'")

# Decode back
print("
Decoded:", enc.decode(token_ids))

# Utility functions for production use
def count_tokens(text: str, model: str = "gpt-4") -> int:
    enc = tiktoken.encoding_for_model(model)
    return len(enc.encode(text))

def truncate_to_budget(text: str, max_tokens: int, model: str = "gpt-4") -> str:
    """Hard-truncate text to fit within a token budget."""
    enc = tiktoken.encoding_for_model(model)
    tokens = enc.encode(text)
    if len(tokens) <= max_tokens:
        return text
    return enc.decode(tokens[:max_tokens])

# Production pattern: token budget management
system_prompt = "You are a helpful assistant specializing in Python."
user_message  = "Explain everything about machine learning. " * 200
context_limit = 8192
output_reserve = 1000

available = context_limit - count_tokens(system_prompt) - output_reserve
safe_message = truncate_to_budget(user_message, available)

print(f"
Original:  {count_tokens(user_message):,} tokens")
print(f"Truncated: {count_tokens(safe_message):,} tokens")
print(f"Fits in context: {count_tokens(system_prompt) + count_tokens(safe_message) + output_reserve} / {context_limit}")

7 Practical Use Cases

API cost management: Enterprise applications processing millions of documents implement hard token limits before every API call. Prompt compression, summarization strategies, and tiered model routing are all built around token budgets. A 10% reduction in average prompt length can translate to millions of dollars in annual savings at scale.

RAG chunk sizing: When splitting documents for Retrieval-Augmented Generation, chunks must be sized in tokens — not words or characters — to fit within the model's context. A 512-token chunk is a common starting point, but must account for the full prompt template including the system message and question.

Multilingual cost optimization: Applications serving multiple languages use token counting to detect high-token-cost queries (Thai, Arabic, Chinese can cost 3-5x more than English for equivalent content) and apply proactive summarization or route to different model tiers.

Prompt injection hardening: Attackers use tokenization boundary tricks — invisible unicode characters or unusual spacing — to split security-relevant words across tokens, evading keyword-based safety filters. Understanding tokenization is essential for building robust input sanitization pipelines.

8 Common Mistakes

❌ Counting words instead of tokens: "1,000 words = 1,000 tokens" is dangerously wrong. English prose averages 1.3 tokens/word. Code runs 2-4 tokens/symbol. Non-English languages can be 3-5x more token-dense. Always count with tiktoken before making cost or context estimates.
⚠️ Using the wrong tokenizer for the model: GPT-4 uses cl100k_base, GPT-3.5 uses p50k_base, Llama uses SentencePiece with its own vocabulary. Counting with the wrong tokenizer gives incorrect results. Always match tokenizer to the exact model being called.
❌ Ignoring special tokens in the context budget: Chat templates add special tokens like start-of-turn and end-of-turn markers that consume budget. A prompt counting as 800 raw tokens may be 950 tokens with the full chat template applied. Always measure after applying the template.
⚠️ Expecting character-level reasoning from token-level models: LLMs see tokens, not characters. "How many r's in strawberry?" fails because the model sees ["straw","berry"] — the individual letter "r" is never an explicit unit of processing. Design tasks and evaluations with token boundaries in mind.
