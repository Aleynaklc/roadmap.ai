1 Simple Explanation

An algorithm is a step-by-step procedure to solve a problem. Think of it like a recipe: given the same ingredients (inputs), following the same steps always produces the same dish (output). In computer science, we care about two things about any algorithm: correctness (does it give the right answer?) and efficiency (how fast does it run, and how much memory does it use?).

Efficiency is measured using Big O notation, which describes how runtime or memory usage scales as the input size grows. An O(n²) algorithm on 1,000 items takes 1 million steps. On 1,000,000 items, it takes 1 trillion steps. Meanwhile, an O(n log n) algorithm on 1 million items takes only ~20 million steps. This difference is the entire reason we study algorithms.

💡 Key insight for AI engineers: Most ML algorithms you'll use are already implemented in optimized libraries. But understanding algorithmic complexity helps you choose between approaches, debug performance issues, and understand why an operation that seems simple is actually slow at scale.

2 Why It Matters for AI Engineers

Algorithms are everywhere in AI. Backpropagation is a graph traversal algorithm. Beam search is a best-first search algorithm. Nearest-neighbor search in vector databases uses approximate search algorithms like HNSW (Hierarchical Navigable Small World). BM25, used in hybrid RAG search, is a ranking algorithm. K-means clustering is an iterative assignment algorithm. If you don't understand algorithmic thinking, you'll treat these systems as magic boxes — and when they break or slow down at scale, you won't know where to look.

3 Real-World Example

Pinecone, Weaviate, and other vector databases use an algorithm called HNSW (Hierarchical Navigable Small World) to find the nearest embeddings to a query vector in milliseconds — even across billions of vectors. A brute-force approach (compare the query to every vector) would take O(n·d) time where d is the embedding dimension. HNSW achieves approximate nearest neighbor search in O(log n) time by building a multi-layer graph of vectors, always skipping toward the answer like "warm/cold" in a game. This is why you can run semantic search over your entire company's document archive in under 50ms.

Big O — Measuring Algorithm Efficiency

python
tokens = ["the", "cat", "sat", "on", "mat"]

# O(1) — Constant time. Fast no matter the size.
first_token = tokens[0]

# O(n) — Linear. Doubles when input doubles.
total_chars = sum(len(t) for t in tokens)

# O(n²) — Quadratic. Dangerous at scale.
# This is why naive self-attention is O(seq_len²)!
similarity_matrix = []
for i in tokens:
    row = [len(set(i) & set(j)) for j in tokens]
    similarity_matrix.append(row)

# O(n log n) — Efficient. Sorting, tree operations.
sorted_tokens = sorted(tokens)   # Timsort — O(n log n)

# O(log n) — Binary search. Used in sorted lookup tables.
import bisect
idx = bisect.bisect_left(sorted_tokens, "on")
print(idx, sorted_tokens[idx])   # finds "on" in log n steps

Recursion & Divide and Conquer

python
# Merge sort — the classic divide-and-conquer algorithm
# Used as the basis for understanding how neural nets split work
def merge_sort(arr: list) -> list:
    if len(arr) <= 1:
        return arr              # base case

    mid = len(arr) // 2
    left = merge_sort(arr[:mid])   # solve left half
    right = merge_sort(arr[mid:])  # solve right half

    # Merge two sorted halves — O(n)
    result, i, j = [], 0, 0
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:
            result.append(left[i]); i += 1
        else:
            result.append(right[j]); j += 1
    return result + left[i:] + right[j:]

scores = [0.92, 0.45, 0.87, 0.23, 0.66]
print(merge_sort(scores))
# [0.23, 0.45, 0.66, 0.87, 0.92] — O(n log n)

Dynamic Programming — Avoiding Redundant Work

python
# Edit distance — measures similarity between two strings
# Used in fuzzy matching, spell checkers, and diff tools
def edit_distance(s1: str, s2: str) -> int:
    m, n = len(s1), len(s2)
    # dp[i][j] = edits needed to turn s1[:i] into s2[:j]
    dp = [[0] * (n + 1) for _ in range(m + 1)]

    for i in range(m + 1): dp[i][0] = i   # delete all of s1
    for j in range(n + 1): dp[0][j] = j   # insert all of s2

    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if s1[i-1] == s2[j-1]:
                dp[i][j] = dp[i-1][j-1]     # no edit needed
            else:
                dp[i][j] = 1 + min(
                    dp[i-1][j],    # delete
                    dp[i][j-1],    # insert
                    dp[i-1][j-1]  # replace
                )
    return dp[m][n]

print(edit_distance("kitten", "sitting"))  # 3
print(edit_distance("gpt4", "gpt-4"))      # 1

Graph Search — BFS and DFS

python
from collections import deque

# BFS — explores level by level. Used for shortest path.
# In AI agents: explore all possible next actions before going deeper
def bfs(graph: dict, start: str) -> list:
    visited, queue, order = {start}, deque([start]), []
    while queue:
        node = queue.popleft()
        order.append(node)
        for neighbor in graph.get(node, []):
            if neighbor not in visited:
                visited.add(neighbor)
                queue.append(neighbor)
    return order

# Example: agent's plan as a dependency graph
task_graph = {
    "search_web": ["summarize"],
    "summarize": ["write_report"],
    "write_report": ["send_email"],
    "send_email": [],
}
print(bfs(task_graph, "search_web"))
# ['search_web', 'summarize', 'write_report', 'send_email']

6 Practical Use Cases

Topological sort for AI pipelines: When you build a LangChain or LlamaIndex pipeline with multiple steps (retrieve → rerank → summarize → respond), you're defining a directed acyclic graph. Topological sort determines the correct execution order, ensuring each step runs only after its dependencies are complete.

Binary search for hyperparameter tuning: When searching for the optimal learning rate between 1e-5 and 1e-1, libraries like Optuna use algorithms similar to binary search to converge on the best value in O(log n) evaluations rather than trying all values.

Greedy algorithms in token generation: "Greedy decoding" in LLMs means always picking the most probable next token — a greedy algorithm. Beam search improves on this by keeping the top-k candidates at each step, exploring more of the solution space without full brute-force search.

7 Common Mistakes

❌ Mistake 1: Nesting loops over large sequences. Two nested loops over a 10,000-token document = 100 million iterations. Always ask: can I vectorize this with NumPy instead?
❌ Mistake 2: Sorting when you only need the top-K items. sorted(scores)[-10:] sorts all n items (O(n log n)). Use heapq.nlargest(10, scores) instead — O(n log k), much faster when k ≪ n.
❌ Mistake 3: Ignoring early-exit conditions in search. If you're checking whether a document contains any of 1,000 keywords, stop as soon as you find the first match — don't check all 1,000.
❌ Mistake 4: Recursion without memoization on overlapping subproblems. Naive recursive Fibonacci is O(2ⁿ). With a cache (@functools.lru_cache), it's O(n). This same principle underlies dynamic programming in sequence alignment.
