1 Simple Explanation

Tool Calling (also called Function Calling) is a capability that allows an LLM to request the execution of external functions during a conversation. Instead of generating only text, the model can output a structured message saying: "I need to call this specific function with these specific arguments." Your application code then executes the function, returns the result to the model, and the model uses that result to continue the conversation.

This transforms LLMs from text generators into reasoning engines that can interact with the real world. Without tool calling, a model can only work with information in its training data or context window. With tool calling, it can check live stock prices, search the web, query your database, send emails, book appointments, control IoT devices, or perform any operation you implement as a function.

The key difference from simple code execution is that the model decides which tool to call, when to call it, and what arguments to pass — based on its reasoning about the user's intent. The developer defines what tools are available, not when they're used. This is the foundation of autonomous AI agents.

💡 Key insight: Tool calling is type-safe by design. You provide a JSON schema that describes each function's parameters. The model is constrained to only call functions you've defined, with arguments that match the schema. This means you can validate tool calls before execution — preventing the model from calling nonexistent functions or passing wrong argument types.

2 Why It Matters for AI Engineers

Tool calling is what makes the difference between a chatbot and an AI agent. Every production AI agent you build — whether it's a customer service bot that can actually check order status, a coding assistant that can run tests, or a data analyst that can query your database — is built on tool calling at its core. Understanding the exact API format, how to structure tool schemas, how to handle parallel tool calls, and how to prevent tool misuse are all critical production engineering skills.

3 Real-World Example

OpenAI's ChatGPT Code Interpreter

ChatGPT's Code Interpreter (now called Advanced Data Analysis) is implemented as a tool. When you upload a CSV and ask "What are the top 10 customers by revenue?", the model doesn't try to answer from its training data. Instead it calls a python_execution tool with code it writes. The tool executes the code in a sandboxed environment and returns the output. The model reads the output and synthesizes a natural language answer, often calling the tool multiple times iteratively — first to load the data, then to analyze it, then to generate a chart. This is exactly the tool-calling loop at work.

4 Step-by-Step Explanation

  1. Define Tools: Write Python functions for your tools. Create JSON schemas that describe each function's name, description, and parameters. The description is critical — the model reads it to decide when to call the tool.
  2. Send to Model: Include the tool schemas in your API call alongside the user message. The model now knows what tools are available.
  3. Model Decision: The model reasons about whether a tool call is needed. If yes, it returns a response with stop_reason: "tool_use" and a structured tool call block.
  4. Execute Tool: Your code reads the tool name and arguments from the response, calls the corresponding Python function, and captures the result.
  5. Return Result: Send the tool result back to the model in a new message with role "tool". The model incorporates this information into its next response.
  6. Loop: If the model needs more tools, repeat. When it has enough information, it generates a final text response.

5 Python Code Example

python
import json
from anthropic import Anthropic

client = Anthropic()

# ── Define tool implementations ────────────────────────────
def get_weather(city: str) -> dict:
    # In production, call a real weather API
    return {"city": city, "temp_c": 22, "condition": "Sunny"}

def search_database(query: str, table: str) -> list:
    # In production, run actual SQL
    return [{"id": 1, "result": f"Mock result for {query} in {table}"}]

TOOLS_MAP = {"get_weather": get_weather, "search_database": search_database}

# ── JSON Schemas (what the model sees) ────────────────────
TOOL_SCHEMAS = [
    {
        "name": "get_weather",
        "description": "Get current weather conditions for a city. Use when the user asks about weather, temperature, or climate.",
        "input_schema": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "City name, e.g. 'London'"}
            },
            "required": ["city"]
        }
    },
    {
        "name": "search_database",
        "description": "Search a database table for records. Use to retrieve specific data.",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {"type": "string"},
                "table": {"type": "string", "enum": ["customers", "orders", "products"]}
            },
            "required": ["query", "table"]
        }
    }
]

# ── Tool calling loop ─────────────────────────────────────
def run_with_tools(user_message: str) -> str:
    messages = [{"role": "user", "content": user_message}]

    while True:
        response = client.messages.create(
            model="claude-opus-4-6",
            max_tokens=1024,
            tools=TOOL_SCHEMAS,
            messages=messages
        )

        # Done — no tool calls needed
        if response.stop_reason == "end_turn":
            return response.content[0].text

        # Execute tool calls and collect results
        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                fn = TOOLS_MAP[block.name]
                result = fn(**block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": json.dumps(result)
                })

        # Add assistant response and tool results to history
        messages.append({"role": "assistant", "content": response.content})
        messages.append({"role": "user", "content": tool_results})

answer = run_with_tools("What's the weather in Paris and do I have any pending orders?")
print(answer)

7 Practical Use Cases

Database query interface: Define a run_sql tool that executes read-only queries against your analytics database. Users ask natural language questions like "How many users signed up last week?" — the model writes the SQL, calls the tool, and returns a synthesized answer with the actual numbers.

Form automation: Define tools for each step of a complex form (validate_address, check_availability, create_booking). The model guides the user conversationally while calling each tool at the appropriate step, handling validation errors gracefully in natural language.

8 Common Mistakes

❌ Vague tool descriptions: The model decides which tool to call based on the description. "Get data" is useless. "Retrieve current weather conditions including temperature and precipitation for a specified city" is clear. Invest as much care in descriptions as in function implementations.
❌ Not handling tool errors: Tools fail — APIs go down, databases timeout, validation fails. If you return an error as a tool result, a good model will adapt and inform the user. If you let the exception propagate and crash the loop, the entire conversation dies.
⚠️ Allowing write operations without confirmation: If a tool can send emails, delete records, or make purchases, require an explicit confirmation step before execution. Models can misinterpret intent. Never allow irreversible actions without a human-in-the-loop confirmation gate.

9 Interview Questions

  • Explain the tool calling flow step by step. What does the model return when it wants to call a tool, and how does your code respond?
  • How do tool descriptions affect model behavior? Give an example of a bad vs. good description for the same tool.
  • What is parallel tool calling? When does a model use it and how does your code handle multiple simultaneous tool calls?
  • How would you prevent a model from calling a dangerous tool (like deleting database records) without user confirmation?
  • What is tool_choice: "required" vs "auto"? When would you force a specific tool call?

10 Mini Practice Tasks

  • Build a 3-tool assistant (calculator, unit_converter, text_word_counter) with full tool calling loop. Test with 5 different user messages that require different tools and verify correct tool selection.
  • Implement error handling: make one of your tools fail 50% of the time randomly. Verify the model handles the error gracefully and informs the user without crashing the loop.
  • Build a parallel tool calling demo: ask "What's the weather in London, Paris, and Tokyo?" and handle the response where the model calls get_weather 3 times simultaneously in one turn.
  • Implement a "confirmation gate": add a send_email tool but intercept calls before execution, print "Model wants to send email to X with subject Y — approve? (y/n)", and only execute if approved.
  • Build a natural language SQL interface: define a run_sql tool that executes SELECT queries on a SQLite database. Test with 10 natural language questions and verify the model writes correct SQL and returns meaningful answers.
