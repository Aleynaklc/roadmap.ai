1 Simple Explanation

Data structures are organized ways to store and access data in memory. Think of them like different types of containers: a list is like a numbered shelf where order matters, a dictionary is like a filing cabinet where you look things up by label, a stack is like a pile of pancakes where you only touch the top, and a queue is like a line at a coffee shop — first in, first out.

In Python, you already use data structures constantly: list, dict, set, and tuple are all built-in. As an AI engineer, you'll also use more specialized structures like heaps, graphs, and trees — often without realizing it, because libraries like NumPy and PyTorch abstract them away for you.

💡 Key insight: Choosing the wrong data structure is one of the most common causes of slow AI code. A lookup that takes O(1) with a dict takes O(n) with a list — and when you're processing millions of tokens, that difference is the difference between a second and an hour.

2 Why It Matters for AI Engineers

Data structures appear everywhere in AI engineering. Token vocabularies are dictionaries. Embedding lookup tables are arrays. RAG pipelines use priority queues for ranked retrieval. Computational graphs in PyTorch are directed acyclic graphs (DAGs). Caches for key-value attention use specialized data structures. Understanding the underlying structure helps you debug bottlenecks, write faster preprocessing pipelines, and reason about what's happening inside your AI libraries.

3 Real-World Example

OpenAI's tokenizer uses a byte-pair encoding (BPE) trie — a tree data structure — to efficiently look up which token IDs correspond to which character sequences. When GPT-4 processes your prompt, it walks this trie millions of times per second. The choice of a trie over a flat list makes tokenization 50–100x faster. Similarly, when Hugging Face's datasets library streams large datasets, it uses memory-mapped arrays — a flat contiguous structure — instead of Python lists, cutting memory usage by 10–100x.

4 Step-by-Step: The Core Structures

Arrays and Lists — Ordered, Indexed Storage

python
import numpy as np

# Python list — dynamic, any type, slower
tokens = ["The", "cat", "sat"]
tokens.append("down")        # O(1) amortized
tokens.insert(0, "start")    # O(n) — shifts everything!

# NumPy array — fixed type, contiguous memory, fast
embedding = np.array([0.1, -0.3, 0.8, 0.2], dtype=np.float32)
# Slicing is O(1) — just a view, no copy
first_half = embedding[:2]

# 2D array — the shape every ML model works with
batch = np.random.randn(32, 512)   # 32 samples, 512-dim embedding
print(batch.shape)   # (32, 512)
print(batch[0])      # first sample — O(1)

Hash Maps (Dictionaries) — O(1) Lookups

python
# Vocabulary lookup — exactly how tokenizers work
vocab = {
    "hello": 1234,
    "world": 5678,
    "<eos>": 2,
    "<pad>": 0,
}

# O(1) lookup — no matter if vocab has 100 or 100,000 words
token_id = vocab.get("hello", vocab["<eos>"])   # 1234

# Frequency counter — common in text preprocessing
from collections import Counter, defaultdict

text = "the cat sat on the mat the cat"
word_freq = Counter(text.split())
print(word_freq.most_common(3))
# [('the', 3), ('cat', 2), ('sat', 1)]

# defaultdict — avoids KeyError when building index
inverted_index = defaultdict(list)
docs = ["ai is great", "python is great for ai"]
for doc_id, doc in enumerate(docs):
    for word in doc.split():
        inverted_index[word].append(doc_id)
# inverted_index['ai'] → [0, 1]

Stacks & Queues — Ordered Processing

python
from collections import deque
import heapq

# Stack (LIFO) — used for DFS, parsing, undo systems
stack = []
stack.append("layer_1")   # push  O(1)
stack.append("layer_2")
top = stack.pop()         # pop   O(1)  → "layer_2"

# Queue (FIFO) — used for BFS, batch processing pipelines
queue = deque()
queue.append("request_1")  # enqueue  O(1)
queue.append("request_2")
first = queue.popleft()   # dequeue  O(1) → "request_1"

# Min-Heap — used for top-k retrieval in vector search
# e.g., find the 5 most similar documents to a query
candidates = [(score, doc_id) for score, doc_id in
              [(0.92, "doc_a"), (0.87, "doc_b"), (0.95, "doc_c")]]
heapq.heapify(candidates)
top_k = heapq.nlargest(2, candidates)
print(top_k)  # [(0.95, 'doc_c'), (0.92, 'doc_a')]

Graphs — How PyTorch Thinks

python
# A computation graph is just a directed graph of operations
# PyTorch builds one automatically during forward pass
import torch

x = torch.tensor([2.0], requires_grad=True)
y = x ** 2 + 3 * x + 1   # builds a graph: y → x²  x → x
y.backward()              # traverses graph backwards
print(x.grad)              # dy/dx = 2x + 3 = 7.0 ✓

# Simple adjacency list graph — for dependency tracking
graph = {
    "embed": [],
    "attention": ["embed"],
    "ffn": ["attention"],
    "output": ["ffn"],
}
# → embed feeds attention feeds ffn feeds output

6 Practical Use Cases

Token caching with LRU: Many LLM inference systems cache KV (key-value) attention states. An LRU (Least Recently Used) cache — implemented with a doubly-linked list + hash map — allows O(1) reads and writes while evicting the oldest entries. This is how systems like vLLM implement KV caching to speed up inference dramatically.

Trie for prefix search: When building an autocomplete feature for an AI-powered text editor, a trie lets you find all completions for a given prefix in O(k) time where k is the prefix length — not proportional to the size of the vocabulary.

Heap for beam search: Beam search (used in text generation to find the best sequence of tokens) uses a priority queue/heap to efficiently track the top-B candidate sequences at each decoding step.

7 Common Mistakes

❌ Mistake 1: Using a Python list for membership checks (if word in word_list). This is O(n). Convert it to a set first for O(1) lookups — critical when processing millions of tokens.
❌ Mistake 2: Concatenating strings inside a loop (result += chunk). Strings are immutable in Python; each concatenation creates a new string. Use a list and "".join() at the end — this is O(n) instead of O(n²).
❌ Mistake 3: Storing large matrices as Python lists of lists instead of NumPy arrays. A 1000×1000 float list uses ~56MB; a NumPy float32 array uses ~4MB — 14× less memory, and math operations are 100× faster.
❌ Mistake 4: Using a sorted list when you need a heap. If you re-sort after every insertion, you're O(n log n) per operation. A heap gives you O(log n) inserts with O(1) min/max access.
