1 Simple Explanation

LlamaIndex (formerly GPT Index) is an open-source data framework specifically designed to connect LLMs with your own data for retrieval and querying. While LangChain is a general-purpose LLM application framework (chains, agents, tools), LlamaIndex is laser-focused on one problem: how do you efficiently index, retrieve, and query over large, heterogeneous data sources?

LlamaIndex introduces several unique abstractions. Data connectors (called "Readers") ingest data from dozens of sources: PDFs, Notion, Google Drive, Slack, SQL databases, GitHub repos, YouTube videos. Indexes structure the ingested data in different ways — a VectorStoreIndex for semantic search, a SummaryIndex for sequential summarization, a KnowledgeGraphIndex for relationship-aware retrieval. Query engines convert natural language questions into retrieval operations against these indexes and return synthesized answers.

💡 Key insight: LlamaIndex vs LangChain is not either/or — they're complementary. LlamaIndex is better for the data ingestion and retrieval layer (RAG infrastructure). LangChain is better for the orchestration and agent layer. Many production systems use both together: LlamaIndex manages the knowledge base, LangChain orchestrates the agentic behavior.

2 Why It Matters for AI Engineers

LlamaIndex handles the hardest part of building production RAG systems: data ingestion at scale. Loading, parsing, cleaning, and chunking data from 20 different file formats is unglamorous but critical engineering work. LlamaIndex does this out of the box with 160+ data loaders. Its advanced retrieval strategies — sub-question decomposition, recursive retrieval, query rewriting — directly address the quality gaps in naive RAG implementations. If your RAG system is returning poor results, LlamaIndex's retrieval primitives are the toolkit to fix it.

3 Real-World Example

Notion's AI Q&A Feature

Notion's AI assistant uses LlamaIndex as the core retrieval infrastructure. When a user asks "What did we decide about the product roadmap in Q3?", LlamaIndex's query engine decomposes this into sub-questions if needed, retrieves the most relevant Notion pages using vector search with metadata filters (date range, workspace), applies re-ranking to improve precision, and synthesizes a response from the retrieved content. The architecture handles millions of documents per workspace with sub-second retrieval latency. LlamaIndex's built-in Notion reader handles the complex Notion API (blocks, databases, nested pages) so the team doesn't need to maintain custom ingestion logic.

4 Step-by-Step Explanation

  1. Load: Use a data connector (SimpleDirectoryReader, PDFReader, NotionReader) to load raw documents into LlamaIndex's Document format.
  2. Transform: Apply node parsers to chunk documents into Node objects. Each Node has text, metadata, and relationships to other nodes (parent document, previous/next chunk).
  3. Index: Insert Nodes into an index. VectorStoreIndex embeds each Node and stores it in a vector store. The index can be persisted to disk and reloaded without re-embedding.
  4. Query: Send a natural language query to a QueryEngine built from the index. The engine retrieves relevant Nodes, passes them with the query to an LLM, and returns a synthesized response with source citations.
  5. Evaluate: Use LlamaIndex's built-in evaluation modules (faithfulness, relevancy, context precision) to measure RAG quality programmatically.

5 Python Code Example

python
from llama_index.core import (
    VectorStoreIndex, SimpleDirectoryReader, Settings
)
from llama_index.llms.openai import OpenAI
from llama_index.embeddings.openai import OpenAIEmbedding
from llama_index.core.node_parser import SentenceSplitter

# 1. Configure global settings
Settings.llm = OpenAI(model="gpt-4o-mini", temperature=0)
Settings.embed_model = OpenAIEmbedding(model="text-embedding-3-small")
Settings.node_parser = SentenceSplitter(chunk_size=512, chunk_overlap=50)

# 2. Load documents from a directory
documents = SimpleDirectoryReader("./docs").load_data()
print(f"Loaded {len(documents)} documents")

# 3. Build vector index (embeds chunks, stores in memory)
index = VectorStoreIndex.from_documents(documents)

# Persist to disk (no re-embedding on next run)
index.storage_context.persist(persist_dir="./storage")

# 4. Create a query engine and ask questions
query_engine = index.as_query_engine(similarity_top_k=5)
response = query_engine.query("What are the main product features?")
print(f"Answer: {response}")
print(f"Sources: {[node.metadata for node in response.source_nodes]}")

# ── Sub-question decomposition for complex queries ─────────
from llama_index.core.query_engine import SubQuestionQueryEngine
from llama_index.core.tools import QueryEngineTool

# Wrap the engine as a tool
tool = QueryEngineTool.from_defaults(
    query_engine=query_engine,
    description="Contains company product documentation"
)

# Sub-question engine breaks complex questions into simpler ones
sub_query_engine = SubQuestionQueryEngine.from_defaults(
    query_engine_tools=[tool]
)

# Complex question gets split into sub-questions automatically
response = sub_query_engine.query(
    "Compare the features introduced in Q1 vs Q2, and explain which had more user impact."
)
print(response)

7 Practical Use Cases

Multi-format enterprise knowledge base: Index your company's Confluence wiki, Slack channels, Google Drive PDFs, and internal SQL databases into a unified LlamaIndex. Employees query once and get answers synthesized from all sources, with citations pointing to the original documents.

Automated RAG evaluation: Use LlamaIndex's built-in FaithfulnessEvaluator and RelevancyEvaluator to programmatically score your RAG system's outputs. Run these evaluations in CI/CD to catch retrieval quality regressions before they reach production.

8 Common Mistakes

❌ Not persisting the index: Rebuilding a VectorStoreIndex from scratch on every application start re-embeds all documents, costing money and adding startup latency. Always persist the index to disk and load it on subsequent runs.
⚠️ Using default chunk sizes for all document types: Code files need smaller chunks (100-200 tokens) to keep functions intact. Legal documents need larger chunks (800-1000 tokens) to preserve argument context. Tune chunk_size and chunk_overlap per document type.
❌ Ignoring source citations: LlamaIndex returns source nodes with every response. Always surface these to users — it enables them to verify answers and builds trust in the system. A RAG system without citations is a hallucination amplifier.
