1 Simple Explanation

RAG — Retrieval-Augmented Generation — is how you give an LLM access to your own documents, databases, or real-time information without retraining the model.

Think of it like this: Instead of expecting a lawyer to memorize every case ever written, you give them a search tool. When a client asks a question, the lawyer searches for relevant cases, reads them, then gives a precise answer. RAG does the same thing for LLMs.

🎯 The core problem RAG solves: LLMs have a knowledge cutoff and can't know about your internal company docs, your product specs, or today's news. RAG bridges this gap without expensive fine-tuning.

2 Why It Matters

RAG is the most commonly deployed LLM architecture in production today. Customer support bots, legal research tools, code assistants with private codebases, medical Q&A systems — all of these use RAG. If you're building an AI product that needs to work with custom data, you need RAG.

3 The Full RAG Pipeline

PHASE 1: INDEXING (run once, or on document updates)

  Your Documents (PDFs, docs, websites, code)
         │
  ┌──────▼──────────────────────────────────┐
  │  CHUNKING  — Split into manageable       │
  │  pieces (e.g., 512 tokens each with      │
  │  50-token overlap)                        │
  └──────┬──────────────────────────────────┘
         │
  ┌──────▼──────────────────────────────────┐
  │  EMBEDDING — Convert each chunk into     │
  │  a dense vector (e.g., 1536 dimensions)  │
  │  using text-embedding-3-small or BGE     │
  └──────┬──────────────────────────────────┘
         │
  ┌──────▼──────────────────────────────────┐
  │  VECTOR STORE — Store embeddings in      │
  │  Pinecone, Weaviate, pgvector, ChromaDB  │
  └─────────────────────────────────────────┘

PHASE 2: RETRIEVAL + GENERATION (every user query)

  User: "What is our refund policy?"
         │
  ┌──────▼──────────────────────────────────┐
  │  EMBED QUERY — Same model as indexing!   │
  │  query_vec = embed("What is our refund…")│
  └──────┬──────────────────────────────────┘
         │
  ┌──────▼──────────────────────────────────┐
  │  SIMILARITY SEARCH — Find top-K chunks   │
  │  via cosine similarity in vector store   │
  │  Returns: top 3-5 relevant passages      │
  └──────┬──────────────────────────────────┘
         │
  ┌──────▼──────────────────────────────────┐
  │  AUGMENT PROMPT — Build context-rich     │
  │  prompt: [System] + [Retrieved docs]     │
  │  + [User question]                        │
  └──────┬──────────────────────────────────┘
         │
  ┌──────▼──────────────────────────────────┐
  │  LLM GENERATION — Model answers based   │
  │  ONLY on provided context                │
  └─────────────────────────────────────────┘
         │
  "Our refund policy allows returns within 30 days..."

4 Complete RAG Implementation

python
# pip install openai chromadb sentence-transformers
from openai import OpenAI
import chromadb
from chromadb.utils import embedding_functions

client = OpenAI()

# ── Step 1: Setup ChromaDB (local vector store) ──
chroma_client = chromadb.Client()
openai_ef = embedding_functions.OpenAIEmbeddingFunction(
    model_name="text-embedding-3-small"
)
collection = chroma_client.create_collection(
    name="docs",
    embedding_function=openai_ef
)

# ── Step 2: Index your documents ──
documents = [
    "Our refund policy: customers can return products within 30 days.",
    "Shipping times: standard delivery takes 5-7 business days in the US.",
    "Premium members get free 2-day shipping on all orders over $50.",
    "Contact support: email support@example.com or call 1-800-SUPPORT.",
    "Holiday hours: our support team is available Mon-Fri 9am-5pm EST.",
]

collection.add(
    documents=documents,
    ids=[f"doc_{i}" for i in range(len(documents))]
)
print(f"✓ Indexed {len(documents)} documents")

# ── Step 3: RAG Query Function ──
def rag_query(question: str, top_k: int = 3) -> str:
    """Answer a question using retrieved context."""

    # Retrieve relevant chunks
    results = collection.query(
        query_texts=[question],
        n_results=top_k
    )
    retrieved_docs = results["documents"][0]

    # Build augmented prompt
    context = "

".join([f"[Doc {i+1}]: {doc}"
                              for i, doc in enumerate(retrieved_docs)])

    system_prompt = """You are a helpful customer support assistant.
Answer questions ONLY based on the provided context.
If the answer isn't in the context, say "I don't have that information."
Never make up information."""

    user_prompt = f"""Context:
{context}

Question: {question}

Answer based only on the context above:"""

    # Generate answer
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_prompt},
        ],
        temperature=0.1  # low temp for factual answers
    )

    answer = response.choices[0].message.content

    # Return answer + sources for transparency
    return {
        "answer": answer,
        "sources": retrieved_docs,
    }

# ── Step 4: Test it ──
result = rag_query("What is your refund policy?")
print("Answer:", result["answer"])
print("
Sources used:")
for s in result["sources"]:
    print(f"  • {s[:60]}...")

5 Advanced RAG Techniques

Chunking Strategies

⚠️ Chunking is the most underrated part of RAG. Bad chunking destroys retrieval quality even with a perfect vector DB. Use semantic chunking, not fixed-size splitting.
python
from langchain.text_splitter import RecursiveCharacterTextSplitter

# Smart recursive splitter that respects document structure
splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,        # ~512 tokens per chunk
    chunk_overlap=64,      # overlap to preserve context across chunks
    separators=["

", "
", ". ", " ", ""]  # try paragraph → sentence → word
)

# Hypothetical Document Embeddings (HyDE) — advanced technique
def hyde_retrieve(question: str, collection) -> list:
    """
    Instead of embedding the question directly,
    ask the LLM to generate a hypothetical answer,
    then embed THAT. Works better for abstract queries.
    """
    hyp_answer = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user",
                   "content": f"Write a factual answer to: {question}"}],
        max_tokens=150
    ).choices[0].message.content

    # Embed the hypothetical answer (not the question)
    results = collection.query(query_texts=[hyp_answer], n_results=5)
    return results["documents"][0]

6 Common Mistakes

❌ Using the same embedding model for indexing and querying is actually correct — but forgetting this leads to silent, hard-to-debug failures.
❌ Not reranking results. Cosine similarity isn't perfect. Add a cross-encoder reranker (like Cohere Rerank or BGE Reranker) to improve precision.
❌ Infinite context window fallacy. Even with 128k context, stuffing too many chunks degrades performance. Keep retrieved context under 4-8k tokens.
